## §6 · Projects & games — in depth

This is the quarry the website CV and one-pagers draw from. Each entry holds more than any one

### Data projects

**US Stock-Market Valuation & Risk** — *real public market data*
- **Question:** across 150+ years of real data, does valuation predict long-run returns, and
  how much downside do you take on by holding equities?
- **Data:** Robert Shiller's price/earnings/CAPE series + CBOE (VIX) + LBMA (gold), pulled with
  a reproducible script. Key cleaning call: recent months store `0.0` as a placeholder for
  not-yet-reported fundamentals — I converted those to missing so they didn't distort the averages.
- **Method:** built a forward 10-year real-return series in SQLite, used a `LAG` window function
  to rank best/worst years, regressed forward returns on starting CAPE with **OLS (statsmodels)**.
- **Found:** higher CAPE → lower forward 10-yr real returns, statistically clear but **R² = 0.11**
  (a real signal, ~a tenth of the variation — not a timing tool); today's CAPE 30.8 is top-5%
  since 1881; real price growth only ~2.4%/yr once inflation is stripped; worst real drawdown
  **−81%** (trough June 1932); stock–gold correlation ≈ 0 (a genuine diversifier).
- **Shipped:** a Python notebook *and* an interactive Tableau Public dashboard on a star-schema
  export (`fact_market_monthly` + `dim_year` + `dim_sector`).
- **Tech:** Python, statsmodels (OLS), pandas, matplotlib, SQL (window functions), SQLite, Tableau.

**Pharmaceutical CRM Sales Analytics** — *synthetic data*
- **Question:** do sales calls drive prescriptions, and where should a rep spend limited time
  across 600 physicians? (A clean, shareable version of my real Viatris CRM work.)
- **Data:** synthetic CRM dataset for 600 physicians from seeded scripts that deliberately inject
  real-world mess (duplicates, missing values, mixed date/text formats, impossible values) so the
  cleaning is real, not decorative.
- **Method:** cleaned and joined call + prescription records (documenting every decision);
  tested the call→script link with correlation + OLS; segmented doctors with **KMeans**; built a
  ranked priority list with `RANK() OVER (PARTITION BY region …)`.
- **Found:** calls move scripts — **r = 0.49** (R² = 0.24), ~**+5.5 scripts per extra call**;
  prescribing is concentrated (top 20% of doctors write ~⅓ of scripts); KMeans isolates a
  **99-doctor "high-potential growth" segment** (of four: Champions 133, Steady mid-tier 51,
  High-potential 99, Low priority 317); rep performance varies sharply (top rep 6,007 scripts
  vs 154 — a coaching/territory signal). Turns "visit more doctors" into a ranked list of 99 to visit first.
- **Shipped:** notebook + Tableau Public dashboard (`fact_scripts` + dim tables).
- **Tech:** Python, scikit-learn (KMeans), statsmodels, SQL (RANK window fns), SQLite, Tableau.

**Free-to-Play Game Economy & Balance** — *synthetic telemetry*
- **Question:** is the economy balanced, which items are broken, and who actually pays?
- **Found:** broadly balanced with mild inflation (sinks recover **~63%** of coins); two

  z-tests); retention **D1/D7/D30 = 47/18/4%**; only **5.8%** ever pay — but predictably so: a
  **logistic model flags likely payers from their first 3 days at AUC 0.80**, so live-ops effort
  can be aimed where it converts.
- **Why it matters for game roles:** the closest of my analyses to live-ops balancing — exploit-

- **Shipped:** notebook + Tableau "Game Economy Health" dashboard.
- **Tech:** Python, logistic regression, proportion z-tests, SQL, Tableau.

**Idle Bank Tycoon — Economy Design Case Study** — *modelled on public estimates, not Kolibri's
internal data*
- **What:** a self-directed economy-design study built as an application case study for a Game

- **Method:** reconstructed the income-vs-upgrade-cost curve over 60 bank levels with **400+ live

## §7 · Voice & personality guide (enforce on every line)

The output has to read like *I* wrote it — a human with one or two opinions — not like polished
AI prose. This is the most important section after the facts.

**Cultivate:**
- **First person with genuine judgement.** Take a position: "I'd ship the ad lever first and
  treat the loan as the bigger retention bet." An LLM doesn't invent opinions like that; it's the
  strongest authenticity signal I have.
- **Concrete nouns over abstractions.** "400+ linked formulas," not "advanced spreadsheet skills."
- **Outcome-first.** Every bullet ends in a result, a number, or a decision.
- **Honest hedging.** "the part I'd defend is the judgement, not the number." Caveats on the
  surface, never buried.
- **Broken rhythm.** Vary sentence length deliberately. A short one lands.
- **Domain words used in context** (sources/sinks, ARPDAU, D1/D7/D30, CAC, entity resolution,
  DCF) — earned by the work, never stuffed for a keyword scan.

**Avoid (these are the AI-writing tells that quietly cost me credibility):**
- The **"X, not Y" antithesis** — it's a verbal tic. Cap it at 2–3 in a whole document.
- **Triads** ("faster, cleaner, smarter") and **tidy closing aphorisms.** Too smooth reads as
  generated.
- **Clichés / résumé-speak:** "single source of truth," "work harder," "end to end," "leverage"
  as filler, "passionate," "results-driven," "proven track record," "thrive in fast-paced," "team
  player," "detail-oriented," "hit the ground running," "synergy." The word "lever" is fine once
  or twice; not ten times.
- **Uniform sentence length** and over-parallelism across paragraphs.
- **Em-dash asides used as a crutch** (I do use them — just not in every sentence).
- **Repeated phrasing between the CV and the letter** — they must read as different registers:
  the CV profile is sharp declarative *positioning*; the letter is warm narrative.

Quick test before sending: read it aloud. If a sentence sounds like every other cover letter,
cut or rewrite it. A human voice with a spine beats flawless balanced prose.

## §10 · How to write the cover letter

**Shape:** ~4 short paragraphs, one page. The structure, in my own logic:

1. **The hook — why *this* company / role, specifically.** Show I actually know the product. A
   concrete observation beats any adjective. ("I've been play-testing Idle Bank Tycoon for a
   while, and somewhere along the way I stopped playing and started taking notes.")
2. **Why I fit — with evidence.** Map the role's core to real work I've done. Lead with the
   strongest, most relevant proof, not a list.
3. **What I'd bring — and the proof I built for it.** Point to the case study / projects: "to put
   that against your game directly, I ran an economy study on Idle Bank Tycoon…"
4. **Close — genuine, brief, no grovelling.** A real reason this role fits the arc of what I do,
   then a plain sign-off.

**Rules:** obey §7 and §8. State enthusiasm through *specifics*, not adjectives — "the rare role
that actually fits me" carries more than "I am passionate about." Don't reuse the CV profile's
phrasing. Map the ad's named requirements to real evidence, but never as a visible checklist.

**Worked example — the Kolibri "Game Economy Designer" letter (the quality bar):**

> Dear Kolibri Games Team,
>
> I've been play-testing Idle Bank Tycoon for a while, and somewhere along the way I stopped
> playing and started taking notes. It reads to me like AdVenture Capitalist rebuilt for a modern
> phone — the same idle backbone, but with a tighter economy, smarter monetization and far more
> polish. Taking a live economy apart like that is genuinely what I do for fun, so when the **Game
> Economy Designer** seat opened on the team, I had to write.
>
> Here's why I think it's the rare role that actually fits me. I've built and balanced two games
> of my own from scratch, and both are economy games at heart: a **Pokémon-style RPG with online
> PvP**, where every stat, move and reward has to stay fair between real players, and **RocketIO**,
> a real-time multiplayer game where a missile economy and constant resource trade-offs *are* the
> game. Building them taught me the same thing over and over — resource management and economy
> design aren't two skills, they're one, and the balance between them is what keeps people playing.
>
> The other half is analytical. Before games I spent three years as a data analyst across equity
> research, pharma and edtech, with a finance and economics core before that — so to Kolibri I'd
> bring the economic read, the data-driven approach and the analyst's toolkit, already proven on
> real game systems rather than in theory. To put that against your game directly, I ran an economy
> study on Idle Bank Tycoon: a **400-formula** model in Excel and Sheets with two live-ops levers,

Skills block** — tailored to the role. Sections, in order:

## §12 · The artifact map (why there are three, and the one-way rule)

| Artifact | Role | Sent? | Profile / Skills? |
|---|---|---|---|
| **This file (`MASTER-CV.md`)** | Single source of truth + voice + guardrails + generator. | No — private | n/a (it's the quarry) |
| **`master-cv.html` → `assets/CV_Meric_Erler.pdf`** | Public "download to see everything" detailed background CV. | Linked on the site | **No, by design** |
| **Per-job one-pager + cover letter** | Tailored to each posting; one page + letter. | **Yes** | **Yes** (tailored) |

**The rule that keeps them from ever contradicting:** facts flow **one way** — from this file →
the website CV → the tailored one-pagers. A one-pager may *emphasise and shorten*, never *differ*.

**Why the website CV has no Profile / Skills:** it is the long, neutral, all-inclusive reference —
the thing a curious recruiter downloads to see the whole record. A profile is a *pitch*, and a
pitch should be aimed at one role; a generic profile on the reference CV would either go stale or
say nothing. Skills read best when grouped *for a role* — which is the one-pager's job. So those
two sections live only on the tailored copy, written fresh each time. (This is also the standing
instruction in §0: don't let an "improve the CV" pass re-add them.)

**To regenerate the downloadable PDF** after editing `master-cv.html`: open it in a browser →
Print → Save as PDF → A4 → save over `assets/CV_Meric_Erler.pdf`. The print stylesheet hides the
nav/footer and forces a clean light theme automatically.
